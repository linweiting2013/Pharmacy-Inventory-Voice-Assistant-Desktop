# 盤點語音助理開發實錄與技術踩坑指南 (Development Log)

本文件紀錄了開發「醫院藥局語音盤點助理」過程中所遇到的各項技術挑戰，特別是與 Google Gemini Live API (預覽版) 整合時所遇到的嚴重底層 Bug，以及我們如何透過架構重構與防呆機制一一克服，最終打造出穩定度 90 分以上、具備臨床可用性的系統。

## 核心挑戰一：1008 Policy Violation 與 1011 Internal Error 頻繁斷線

### 🛑 問題描述
在早期開發中，系統經常在藥師回報數量後，或是助理試圖呼叫 `update_drug_quantity` (更新試算表) 的瞬間，毫無預警地崩潰。WebSocket 伺服器會強制斷線並拋出 `1008 Policy Violation` 或 `1011 Internal Error`，導致盤點流程完全中斷。

### 🔍 原因分析
經過反覆測試與架構梳理，我們發現這是 Google Gemini Preview API 在處理 **「非同步/非阻塞 (NON_BLOCKING) 函式呼叫 (Function Calling)」** 時的底層伺服器狀態機 (State Machine) 缺陷。
當模型處於生成音訊 (Speaking) 或持續聆聽的狀態時，若我們回傳了 `FunctionResponse` 給伺服器，伺服器的排程器會發生競態條件 (Race Condition) 或狀態衝突，進而導致伺服器崩潰。

### 💡 解決方案 (架構演進)
1. **嘗試「射後不理 (Fire-and-forget)」 (治標不治本)**：
   我們一度將 `voice_agent.py` 中的 `NON_BLOCKING` 工具呼叫設定為「不回傳 FunctionResponse」，試圖騙過伺服器。雖然這減少了 1011 錯誤，但當模型需要確認複雜資訊時，仍會誘發 1008 錯誤。
2. **最終解法：回退至同步阻塞 (Synchronous Blocking) 與切換穩定模型**：
   我們徹底移除了工具設定中的 `"behavior": "NON_BLOCKING"`，並將模型切換至智商與穩定度雙高的 **`gemini-3.1-flash-live-preview`**。當助理呼叫工具時，語音串流會短暫暫停，等我們 Python 執行緒把資料寫入 Google Sheets 後，再把 `FunctionResponse` 回傳給伺服器。這徹底消除了 1008/1011 斷線問題！

## 核心挑戰二：模型幻覺 (Hallucination) 與參數型態驗證崩潰

### 🛑 問題描述
當藥師不小心口誤（例如說出：「二十五和七片，六片」），助理有時候不會開口詢問，而是直接觸發 `1011 Internal Error` 死亡。
另外，若使用 `gemini-2.5-flash-native-audio-preview` 模型，有時會陷入無限的自言自語迴圈，狂念「掌聲鼓勵 播放音效 掌聲鼓勵」。

### 🔍 原因分析
1. **幻覺迴圈**：2.5-native 模型在處理原生音訊與複雜的邏輯打斷時，穩定性極差，非常容易進入無限音訊生成的幻覺狀態。
2. **型態驗證崩潰**：當藥師講中文字 (例如「七片」)，模型有時候在背景呼叫工具時，會把 `"blister": "七"` (字串) 塞進原本定義為 `INTEGER` (整數) 的欄位。Gemini 伺服器底層的 JSON Schema 驗證器遇到型態不符，沒有優雅地回傳錯誤，而是直接核心當機並吐出 1011。

### 💡 解決方案
1. **防呆 Schema 定義**：我們在 `voice_agent.py` 的工具參數描述 (Description) 中，強制加上了 **「必須是純數字」** 的嚴格限制。
2. **行為約束 (System Prompt)**：在 `gui.py` 的系統提示詞中，我們新增了「口誤更正」的應對守則：
   > 「若藥師口誤或要求更正數量，請確認最終數量後再呼叫工具。呼叫工具時，**所有數量參數必須嚴格使用「純阿拉伯數字」**(例如 7，絕對不可使用國字如「七」或「空」)。」
3. **未定義工具防護**：加入針對未知工具 (Unknown Tool) 的 `else` 判斷分支，防止模型自行發明工具名稱而導致程式靜默失敗。

## 核心挑戰三：喇叭回音導致的「自我中斷」 (Self-Interruption)

### 🛑 問題描述
如果藥師不戴耳機，而是使用電腦喇叭外放助理的聲音，麥克風會把助理自己的聲音收進去。助理聽到聲音後，會以為藥師正在「插嘴打斷」，於是立刻停止說話，導致永遠講不完一句話。

### 💡 解決方案 (軟體級半雙工防回音)
我們在 `voice_agent.py` 與 UI 介面中實作了 **「防喇叭回音 (半雙工) 模式」**。
- 當系統正在播放音訊佇列 (`audio_queue_output`) 的內容時，立起 `is_speaking = True` 旗標。
- 只要旗標為 True，從麥克風收到的音訊幀 (`msg`) 就會被丟棄 (Mute)，不傳給伺服器。
- 音訊播放完畢後，故意等待 `0.2 秒`，讓實體空間中的「喇叭餘音」消散，再重新打開麥克風收音。
此舉完美解決了不戴耳機時的自我打斷問題。

## 核心挑戰四：多人多台同時盤點不同調劑台

### 🛑 問題描述
醫院藥局有多個調劑台，負責不同區塊的藥師會同時進行盤點，但原先的 Google Sheets API 寫死只讀取第一張分頁 (`sheet1`)，無法平行作業。

### 💡 解決方案
我們修改了 `sheets_client.py` 與 `gui.py` 的架構：
1. 開發 `get_worksheets` 靜態方法，使用 `gspread` 連線並動態抓取試算表內的所有分頁名稱。
2. 在「系統設定」介面實作「取得工作表清單」按鈕與下拉選單。
3. 利用 `config.json` 記住每台電腦負責的「指定工作表」。
如此一來，不同的電腦與使用者都能共用同一份程式與 API 金鑰，但在啟動時各自綁定不同的分頁，實現完美的併發作業。

## 核心挑戰五：醫療免責聲明與遺漏資訊

### 🛑 問題描述
1. 由於助理是基於醫療/安全守則嚴格訓練的 LLM，當它說出真實藥名時，經常會在句尾加上冗長的「本資訊僅供參考，請諮詢專業醫師...」免責聲明。
2. 藥師新增「未列帳藥品」時，原本的程式缺少了「代碼、儲位、包裝單位」等重要欄位。

### 💡 解決方案
1. **解除緊箍咒**：在系統提示詞的「嚴格禁止事項」加入一條：*「嚴禁加上任何醫療免責聲明...因為你現在是醫院內部盤點系統」*。一擊必殺，助理立刻變得乾脆俐落。
2. **補齊 Context 視野**：修改 `load_sheet_data` 時傳遞給大腦的 Context 字串，將「代碼」欄位補上；並在 System Prompt 賦予助理在新增藥品時「主動且依序詢問四個空缺欄位，不知則留空」的 SOP。

---

## 結論與測試結果
經歷上述的架構迭代與 Prompt 工程，目前系統已具備極高的強健性：
- **反應速度**：切換為阻塞模式與 `gemini-3.1` 後，回應時間依然能壓在體感可接受的毫秒級範圍。
- **對話流暢度**：容許結巴、口誤、中途改變主意，模型具備優異的上下文整合能力。
- **資料正確性**：透過防呆 Schema 徹底消滅了填表格式錯誤與連線當機。
本專案成功證明了使用先進語音 LLM 替代傳統雙人盤點人力的可行性。
